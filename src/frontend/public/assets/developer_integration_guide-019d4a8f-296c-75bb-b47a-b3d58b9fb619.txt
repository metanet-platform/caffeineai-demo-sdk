Developer Integration Guide
Welcome, developer! This guide provides everything you need to integrate your application with the Metanet.page platform. Integration allows your app to interact with the user's wallet, create posts, and much more.

Apps on the Metanet.page platform run in a secure iframe environment, ensuring that your app can interact with the user's wallet without the user exposing sensitive information. This guide will walk you through the core messaging protocol, how to use our JavaScript SDK, and best practices for building robust applications. The sandboxed iframe only gives allow-script and allow itsself as origin permissions, ensuring a secure environment. No alerts, confirmations, or prompts are used, as all interactions are handled through the messaging protocol.

The core of our integration is a simple and robust messaging protocol based on window.postMessage. This makes it language-agnostic, allowing you to build apps in vanilla JavaScript, React, Vue, or any other framework that can run in a browser. For convenience, we also provide a lightweight JavaScript SDK to simplify these interactions.

The Core Protocol
Communication between your app (loaded in an iframe) and the Metanet.page host platform happens via messages. You don't need any special libraries, just the standard browser APIs.

Sending Messages to the Platform
To send a message, you use window.parent.postMessage. Every message must be an object with a specific structure:

window.parent.postMessage({
  command: "ninja-app-command", // This is always the command
  detail: { 
    type: "connection", // The type defines the action
    // ...other properties for this type
  }
}, "*");
command: Must always be the string "ninja-app-command". This identifies the message as intended for our platform.
detail: An object containing the actual instructions.
type: A string that specifies the action you want to perform (e.g., "connection", "pay").
Additional properties depend on the type and are detailed in the Command Reference.
Receiving Messages from the Platform
The app will load inside the iframe appending all query parameters from the URL to the window.location.search. The platform automatically adds a metanetLng parameter with the user's language preference (e.g., ?metanetLng=en). If the user is anonymous, the platform will send a connection-response message with an anonymous: true property and no wallet property. If the app is being loaded from a Premium Username, the connection-response payload will also include appPageSchema containing the Username Schema object that can be found in any Premium username's Verified Card by clicking the Verified blue badge. This schema can be used by the app to identify where it was loaded from and access profile information.

Apart the query on app load, the platform will respond to the app's commands as descrived further below.

To receive responses, you listen for message events on the window object. It's crucial to validate that the message is from the platform by checking its structure.

window.addEventListener("message", (event) => {
  // IMPORTANT: Add origin check in production for security
  // if (!["https://www.metanet.page","https://www.metanet.ninja"].includes(event.origin)) return;

  const { command, type, payload } = event.data;

  if (command === "ninja-app-command") {
    console.log("Received message from platform:", { type, payload });

    if (type === "connection-response") {
      // Handle the connection response
    } else if (type === "pay-response") {
      // Handle the payment response
    }
    // ... and so on for other response types
  }
});
JavaScript SDK (Convenience Wrapper)
To make development faster, you can use our lightweight JavaScript SDK. It's a simple wrapper around the Core Protocol that provides two main functions: sendCommand and onCommand. It handles the postMessage structure and event listening for you.

This is not a package to install. Simply include this script in your app's HTML or main JavaScript file.

<script type="module">
  // Create a shared event target to manage commands within your app
  const eventTarget = new EventTarget();

  // Wraps window.parent.postMessage with the required structure
  const sendCommand = (commandObj) => {
    try {
      window.parent.postMessage({ 
        command: "ninja-app-command", 
        detail: commandObj // commandObj is { type: "...", ... }
      }, "*");
      // Also dispatch locally for your app's own listeners
      eventTarget.dispatchEvent(new CustomEvent('platform-command', { detail: commandObj }));
    } catch (err) {
      console.error('Error sending command:', err);
      throw err;
    }
  };

  // A listener that centralizes responses from the platform
  const onCommand = (callback) => {
    window.addEventListener('message', (event) => {
      // Optional: Check event.origin for security
      if (event.data && event.data.command === "ninja-app-command") {
        callback(event.data); // callback receives { command, type, payload }
      }
    });
  };

  // Expose the SDK to your app
  window.platformSDK = { sendCommand, onCommand };
</script>
With this script included, you can access the functions via window.platformSDK.sendCommand(...) and window.platformSDK.onCommand(...).

Typical Integration Flow
A standard interaction between your app and the platform follows these steps:

Establishing Connection: On launch, your app sends a message with type: "connection". The platform replies with a "connection-response" message containing the user's wallet details and other essential data. This data should be stored and used to verify subsequent messages.
Requesting an Action (e.g., Payment): To initiate a payment, your app sends a message with type: "pay" and a payload detailing the recipients and amounts.
Handling the Response: The platform processes the request (which may require user confirmation) and replies with a "pay-response" message. Your app must check this response to confirm if the payment was successful.
Submitting Content: Suppose you wanted to change the user for the tools you provided and now you are ready to let them post the produced content on their Feed. After any required payment authorisations are confirmed, you can broadcast or keep for later, then your app can send a message with type: "create-post" to publish content. The platform will confirm the result with a "create-post-response".
Command Reference
This section details the type and detail object for each message you can send, and the payload for each response you can receive.

Connection & Authentication
type: "connection" (App → Platform)
Send on app launch to establish the connection. The optional navbg field can suggest a background color for the platform's navigation bar.

// Message detail object
{
  type: "connection",
  navbg?: string // Optional: preferred navigation background color
}
type: "connection-response" (Platform → App)
The platform's response to a connection request. Contains credentials for validating future messages. We show the full structure here, but you only need to use the payload or in this case exceptionally also the icIdentityPackage - for all other types you only need the response's payload.

// Response full object
{
  command: "ninja-app-command",
  type: "connection-response",
  payload:{
    appId: string,
    timestamp: number, // Unix timestamp in milliseconds
    anonymous: true, // this is present and set to true when the user is not logged in
    wallet: {
      address: string, // this is dereived from the publicKeyHex
      publicKeyHex: string, // Hex-encoded public key, this can be verified against response signature
      rootPrincipal: string, // User's root ICP principal
    }
    icDelegation: JSON,   // Public delegation included in the signed payload (time-bounded)
  }
  appPageSchema: {  // Present when loaded from a Premium Page
    "protocolId": "NINJA-NS",
    "symbol": "authenticator",
    "...": "...",       // Other schema fields
    "properties": {
      "username": "Authenticator", // Important: The Premium Username
      "profile": {
        "bio": "This is the official Authenticator app...",
        "...": "..."    // Other profile fields
      },
      "pubkey": "0390247c087a3ae15504d2beadf6d8f137ab970ca3215197ce30864f26f7ab93ab",
      "signature": "304402201de7751496743f56a4c4713117f532301f6a4c9b2cc635225e7145118ba07c1a02203469738f3299ec0b78aeaa3438156336f7aa3f7d0b28a3a7d51d3ce487038d85",
      "...": "..."      // Other properties
    }
  },
  genericUseSeed: string, // Fixed 32-byte hex seed for app-specific cryptographic operations
  icIdentityPackage: {   // Full delegation package including private key 
    delegation: JSON,
    privateKey: string
  },
  signature: "dummy_signature" // Signature of the payload only
}
Key Field Explanations
icDelegation: A time-bounded delegation for ICP operations. This delegation expires periodically for security and should not be stored long-term. Use it for immediate ICP blockchain operations.
rootPrincipal: The user's stable ICP principal identifier. Do not trust this piece of info. This only tells the logged in user claims to own that principal. But there is no guarantee this is true in this version of the SDK. Use it as certification of the user approving this Principal, potentially to receive ICP tokens in it, but not as proof of ownership of that address.
genericUseSeed: A fixed 32-byte hex-encoded seed derived specifically for your app. This seed is deterministically generated from the user's identity and your app ID, providing a consistent cryptographic foundation for app-specific operations like key derivation, encryption, or signing. Unlike the time-bounded icDelegation, this seed remains constant for the user-app pair and can be safely stored for persistent cryptographic operations.
appPageSchema: Only present when the app is loaded from a Premium Username page. Contains the complete Username Schema object including properties.username (the Premium Username) and properties.signature. For verification, validate the signature against all properties except the signature itself using properties.pubkey. This allows your app to authenticate which Premium Username it was loaded from.
Payments (BSV, ICP & KDA)
type: "pay" (App → Platform)
Requests a payment. Supports BSV, ICP token, and KDA payments. Each item in recipients can be a reason-only object for platform fees, or a full payment object.

BSV Payments (Default)
// BSV Payment (existing behavior)
{
  type: "pay",
  ref: "unique_reference", // A unique ID you create to track this request
  recipients: [
    { reason: "AI_IMG" }, // Platform fee for a specific service
    { 
      address: "target_address", // Bitcoin address
      value: 330000, // in satoshis OR
      fiatValue: 12, // in fiat currency (platform will convert)
      currency: "USD", // optional, defaults to USD. Supports: USD, EUR, GBP, JPY, AED, AUD, CAD, CHF, CNY, INR, BRL, MXN, KRW, RUB, ZAR, SEK, NOK, DKK, SGD, HKD, NZD, THB, IDR, MYR, PHP, TWD, PLN, TRY, ARS, CLP, COP
      note: "optional_note"
    }
  ]
}
Important Currency Rules:
Single currency only: All recipients with fiatValue must use the same currency. Mixed currencies will produce an error.
Auto-conversion: If any recipient specifies a currency, ALL recipients will display their fiat values in that currency to the user (even if they specified values in satoshis only).
Default: If no currency is specified, USD is used by default.
ICP Token Payments (New)
For ICP token payments, include a token object with protocol specification. Currently supports single recipient only.

// ICP Token Payment
{
  type: "pay",
  ref: "unique_reference",
  token: {
    protocol: "ICP",
    specification: {
      ledgerId: "choose_from_supported_ledger_ids" // Canister ID of the token ledger
    }
  },
  recipients: [
    {
      address: "target_principal", // Principal ID
      value: 1000000, // Amount in token's smallest unit (e.g., decimal with up to 8 decimal digits for ICP)
      note: "Payment for premium service"
    }
  ]
}
Supported ICP Ledger IDs:

ryjl3-tyaaa-aaaaa-aaaba-cai - ICP
xevnm-gaaaa-aaaar-qafnq-cai - ckUSDC
6c7su-kiaaa-aaaar-qaira-cai - GLDT
mxzaz-hqaaa-aaaar-qaada-cai - ckBTC
ss2fx-dyaaa-aaaar-qacoq-cai - ckETH
KDA (Kadena) Payments
For Kadena payments, set protocol: "KDA" in the token object. Currently supports single recipient only. The recipient address must be a Kadena k: account.

// KDA Payment
{
  type: "pay",
  ref: "unique_reference",
  token: {
    protocol: "KDA",
    specification: {
      chainId: "2" // Optional, defaults to "2". Kadena chain ID (0-19)
    }
  },
  recipients: [
    {
      address: "k:abc123...", // Kadena k: account
      value: 1.5, // Amount in KDA (decimal)
      note: "Payment for service"
    }
  ]
}
KDA Payment Notes:
Amount format: KDA amounts are in decimal (e.g., 1.5 = 1.5 KDA), unlike ICP which uses smallest units.
Chain ID: Defaults to chain 2. Most KDA operations on the platform use chain 2.
Account format: Recipient must be a valid Kadena k: account (prefixed with "k:").
Single recipient: Only one recipient per payment request is supported.
type: "pay-response" (Platform → App)
Indicates the outcome of a payment request. Match it using the ref you provided. Response format varies depending on payment type (BSV, ICP, or KDA).

BSV Payment Response
// BSV Payment Success
{
  ref: "unique_reference",
  success: true,
  rawTxHex: "transaction_hex", // Bitcoin transaction hex
  responseCode: "OK_SUCCESS",
  timestamp: number, // Unix timestamp in milliseconds
}

// BSV Payment Failure
{
  ref: "unique_reference",
  success: false,
  rawTxHex: null,
  message: "Error message",
  responseCode: "ERR_ABORTED",
  timestamp: number, // Unix timestamp in milliseconds
}
Broadcasting BSV Transactions
When you receive a successful BSV payment response, you get the rawTxHex - the signed transaction ready for broadcast. While you can broadcast this at your own pace, we strongly recommend using our overlay service for indexing and UTXO management. Please be aware we do NOT index transactions bigger than 33Kb.

Broadcasting API Call:
// Step 1: Create the payload
const payload = {
  data: {
    action: "broadcastTransactions",
    raws: [rawTxHex], // Array of transaction hex strings (single or multiple)
    params: { source: "your-app-id", timestamp: Date.now() }
  }
};

// Step 2: Sign the payload for authentication
// You'll need elliptic and crypto-js libraries for signing
import { ec as EC } from 'elliptic';
import SHA256 from 'crypto-js/sha256';

const ecInstance = new EC('secp256k1');
const keyPair = ecInstance.keyFromPrivate(yourPrivateKeyHex);
const publicKeyHex = keyPair.getPublic(true, 'hex');

// Create canonical payload string (wrap data in { data: ... })
const canonicalPayloadStr = JSON.stringify({ data: payload.data });

// Hash and sign (CRITICAL: Buffer.from without 'hex' parameter!)
const hashHex = Buffer.from(SHA256(canonicalPayloadStr).toString());
const signature = keyPair.sign(hashHex, { canonical: true }).toDER('hex');

// Step 3: Make the API call
const response = await fetch('https://api.metanet.ninja/data/api', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'x-signature': signature,      // DER-encoded signature
    'x-pubkey': publicKeyHex       // Compressed public key (33 bytes, hex)
  },
  body: canonicalPayloadStr        // Send the canonical string (with data wrapper)
});

const result = await response.json();

// Response format from Metanet API:
// {
//   success: true,
//   data: [
//     {
//       txid: "abc123...",           // Transaction ID
//       success: true,               // Broadcast status boolean
//       error: null                  // null if successful, error object if failed
//     },
//     // ... one object per transaction in raws array
//   ]
// }
//
// Error response:
// {
//   success: false,
//   error: "Error message describing what went wrong"
// }
Response Details:
Success: Returns array of results, one per transaction
txid: The transaction ID if broadcast was successful
status: "success" for successful broadcasts
error: null on success, error object with details on failure
Authentication Requirements:
x-signature: DER-encoded ECDSA signature of SHA256(canonicalPayloadStr)
x-pubkey: Compressed public key (33 bytes hex) matching the signature
canonicalPayloadStr: JSON.stringify({ data: payload.data }) - wraps data object
Body: Send canonicalPayloadStr (same as what you signed)
Hash: Buffer.from(SHA256(canonicalPayloadStr).toString()) - NO 'hex' parameter!
Benefits: Our overlay service automatically indexes UTXOs, tracks transaction states, and provides better error handling. It uses the BSV P2P network with fallback mechanisms for reliable broadcasting.

Content Creation & Swaps
type: "create-post" (App → Platform)
Submits content to be published by the app. Ensure all media files are correctly referenced. Do not confuse this payload with the response payload. This payload key is simply a key required for this specific command type "create-post"

// Message detail object
{
  type: "create-post",
  params: {
    headline: "text",
    nftDescription: "text",
    previewAsset: {type:"image", file:File, preview:"dataURL"},
    extraContents: [{type:"image", file:File, preview:"dataURL"},{type:"long_text", contentData:"text"}],
    appQuery: { campaign: "cname", tstamp: Date.now(), ref: "internal reference" },
    appEmbed: { url: "iframe_url", type: "video|audio|shop|game", shape: "square|landscape" }
  }
}
Parameter Details
appEmbed:
Embeds an interactive iframe in the post. Perfect for video players, audio players, mini-games, or shop widgets.

url - The iframe source URL (required)
type - Display style: "video" (play icon), "audio" (music icon), "shop" (cart icon), "game" (gamepad icon). Defaults to "video"
shape - Aspect ratio: "square" (1:1), "landscape" (9:6). Defaults to "square"
previewAsset:
A thumbnail/preview image for your post. Must be provided as an object with type, file, and preview properties.

Example: Creating previewAsset from URL
To use an image from a URL as your preview, you need to fetch it, convert to a File object, and create a dataURL:

// Fetch image from URL
const response = await fetch(thumbnailUrl);
const blob = await response.blob();

// Convert Blob to dataURL
const dataURL = await new Promise((resolve, reject) => {
  const reader = new FileReader();
  reader.onloadend = () => resolve(reader.result);
  reader.onerror = reject;
  reader.readAsDataURL(blob);
});

// Convert dataURL to File
const dataURLtoFile = (dataurl, filename) => {
  const arr = dataurl.split(',');
  const mimeMatch = arr[0].match(/:(.*?);/);
  if (!mimeMatch) throw new Error("Invalid data URL");
  const mime = mimeMatch[1];
  const bstr = atob(arr[1]);
  let n = bstr.length;
  const u8arr = new Uint8Array(n);
  while (n--) {
    u8arr[n] = bstr.charCodeAt(n);
  }
  return new File([u8arr], filename, { type: mime });
};

const imageFile = dataURLtoFile(dataURL, 'thumbnail.jpg');

// Create previewAsset object
const previewAsset = {
  type: 'image',
  file: imageFile,
  preview: dataURL
};

// Use in create-post
sdk._sendCommand({
  type: 'create-post',
  params: {
    headline: 'Check out this video!',
    nftDescription: 'Amazing content',
    previewAsset: previewAsset,
    appEmbed: {
      url: 'https://example.com/embed/video123',
      type: 'video',
      shape: 'square'
    }
  }
});

type: "authorise-swap" (App → Platform)
Creates a swap offer for an NFT you own. This generates a partially signed transaction that allows another user to purchase your NFT for a specified amount. Used in NFT marketplace scenarios with STAS-789 tokens.

{
  type: "authorise-swap",
  contractTx: string,    // NFT contract transaction ID
  value: number          // Amount in satoshis you want for the NFT
}
How NFT Swaps Work
The STAS-789 swap protocol enables atomic NFT trades:

Seller (authorize-swap): Creates a partially signed offer transaction
Buyer (swap-buy): Completes the transaction by adding payment inputs
Atomic: Either both parties get what they want, or the transaction fails
Platform Role: Validates NFT ownership, creates swap tx, handles confirmations
Requirements: You must own exactly one unspent UTXO of the NFT. The platform verifies ownership before creating the swap offer. Wallet authentication required.
type: "authorise-swap-response" (Platform → App)
Response after creating a swap offer. Contains the partially signed swap transaction hex.

// Success response
{
  timestamp: string,
  success: true,
  contractTx: string,        // NFT contract transaction ID
  parsedNFT: {              // Parsed NFT metadata from STAS-789
    protocolId: "STAS-789",
    symbol: string,          // NFT symbol
    tokenId: string,         // Unique token identifier
    // ... other NFT properties
  },
  wanted: {
    satoshis: number         // Amount requested for NFT
  },
  swapHex: string,          // Partially signed swap transaction (hex)
  responseCode: "OK_SUCCESS"
}

// Failure responses
{
  timestamp: string,
  success: false,
  error: string,
  responseCode: 
    "ERR_NOT_FOUND" |      // Contract transaction not found
    "ERR_NO_MARKER" |      // Invalid STAS-789 format
    "ERR_PARSE_FAIL" |     // Failed to parse NFT metadata
    "ERR_NO_UTXO" |        // NFT not found in wallet or already spent
    "ERR_USER_CANCEL"      // User rejected the swap offer
}
Next Step: Share the swapHex with potential buyers. They use this hex with swap-buy to complete the purchase.

type: "swap-buy" (App → Platform)
Completes an NFT purchase using a swap offer from the seller. Requires the swap hex from the seller's authorize-swap response.

{
  type: "swap-buy",
  swapHex: string,       // Partially signed swap tx from seller
  contractTx: string,    // NFT contract transaction ID
  value: number          // Amount in satoshis (must match seller's asking price)
}
Swap Buy Process
Platform validates the swap offer and NFT ownership
Fetches required UTXOs from your wallet for payment + fees
Consolidates UTXOs if needed (may require additional transaction)
Completes the partially signed swap transaction with your payment inputs
Verifies NFT transfer and broadcasts the final transaction
NFT transfers to your wallet, satoshis transfer to seller
Requirements: Sufficient BSV balance to cover the NFT price plus transaction fees. The platform automatically handles UTXO consolidation and fee calculation. Wallet authentication required.
Security: Platform validates that the seller still owns the NFT at the moment of purchase. If the NFT was already sold or spent, the transaction will fail with an error.
type: "swap-buy-response" (Platform → App)
Response after attempting to purchase an NFT through a swap.

// Success response
{
  timestamp: string,
  success: true,
  contractTx: string,        // NFT contract transaction ID
  swapTxHex: string,        // Final completed swap transaction hex
  responseCode: "OK_SUCCESS"
}

// Failure responses
{
  timestamp: string,
  success: false,
  error: string,
  responseCode: 
    "ERR_SWAP_EXEC" |      // Error executing swap (check error message)
    "ERR_NO_MARKER" |      // Invalid STAS-789 format
    "ERR_PARSE_FAIL" |     // Failed to parse NFT metadata
    "ERR_USER_CANCEL" |    // User rejected the purchase
    "ERR_UNKNOWN"          // Unexpected error
}
Common Error Scenarios:

Insufficient funds: Not enough BSV to cover NFT price + fees
NFT already sold: Seller's NFT UTXO was spent by another buyer
Invalid swap offer: Swap hex is malformed or expired
Seller = Buyer: Cannot buy your own NFT
Consolidation failed: Unable to prepare payment UTXOs
Utilities & Simple Commands
type: "open-link" (App → Platform)
Requests to open a specific URL. Ensure the URL is correctly formatted.

// Message detail object
{
  type: "open-link",
  url: "https://..."
}
💡 Pro Tip: Auto-Intercept Link Clicks

You can automatically convert all external link clicks in your app to use the open-link command. This gives users a confirmation dialog before leaving your app.

Add this to your app's initialization:

document.addEventListener('click', (e) => {
  const link = e.target.closest('a');
  if (!link || !link.href) return;
  
  const url = link.href;
  const currentOrigin = window.location.origin;
  const linkOrigin = new URL(url).origin;
  
  // Only intercept external links
  if (linkOrigin !== currentOrigin) {
    e.preventDefault();
    sendCommand({ type: 'open-link', url: url });
  }
}, true);
✓ Works for all links in your app
✓ Internal navigation works normally
✓ External links show user confirmation

What is BUMP?
BUMP (Bitcoin Unified Merkle Path) is a standardized format for Merkle proofs that enables lightweight SPV (Simplified Payment Verification) without requiring a full blockchain node.

Use Cases: Verifying transactions, submitting to overlay services, building SPV wallets, cross-chain bridges, or any scenario requiring cryptographic proof of transaction inclusion in a block.

🎉 Beta Period: Full transaction retrieval with BUMP proofs is currently free during the beta period. Take advantage of this to build and test your SPV applications!
Authentication Required: This endpoint requires wallet authentication. Unauthenticated requests will prompt the user to connect their wallet.

Geolocation
type: "geolocation" (App → Platform)
Requests the user's geolocation with high accuracy tracking. The platform will continuously monitor location accuracy and send updates as precision improves, with a target accuracy of 100 meters.

{
  type: "geolocation",
  ref: "unique_ref"      // Unique reference to track this request
}
How Geolocation Works
Progressive Accuracy: The platform starts with available location data and continuously improves precision
Multiple Updates: Your app will receive multiple responses as accuracy improves (isFinal: false)
Final Response: When accuracy reaches ≤100m or after timeout, a final response is sent (isFinal: true)
Heartbeat: Updates sent every 30 seconds even if accuracy hasn't improved
Automatic Cleanup: Location tracking stops automatically after final response or if app sends a new geolocation request
Permission Required: This requires browser geolocation permission. The user will be prompted if permission hasn't been granted yet.
type: "geolocation-response" (Platform → App)
Response containing geolocation data. Your app may receive multiple responses for a single request as accuracy improves.

// Progressive update (accuracy improving)
{
  ref: "unique_ref",
  timestamp: number,
  location: {
    latitude: number,      // Latitude in degrees
    longitude: number,     // Longitude in degrees
    accuracy: number       // Accuracy in meters (e.g., 450.5)
  },
  isFinal: false,         // More updates coming as accuracy improves
  responseCode: "OK_SUCCESS"
}

// Final response (target accuracy reached)
{
  ref: "unique_ref",
  timestamp: number,
  location: {
    latitude: number,
    longitude: number,
    accuracy: number       // ≤100 meters
  },
  isFinal: true,          // No more updates for this request
  responseCode: "OK_SUCCESS"
}

// Error response
{
  ref: "unique_ref",
  timestamp: number,
  error: string,
  isFinal: true,
  responseCode: "ERR_NOT_SUPPORTED" // or "ERR_PERMISSION_DENIED"
}
Best Practice: Listen for multiple responses and update your UI as accuracy improves. Check the isFinal flag to know when location tracking has stopped.

Response Codes & Meanings
OK_SUCCESS: Operation completed successfully.
ERR_POST_FAIL: Post creation failed due to incorrect input or errors.
ERR_POST_EXCEPTION: Exception occurred during posting.
ERR_ABORTED: Operation was aborted by the user or process.
ERR_UNKNOWN: An unknown error occurred.
ERR_REJECTED: The request was rejected by the host platform.
ERR_NO_DATA: QR scanner couldn't detect any readable data.
ERR_KDA_PREP_FAILED: Failed to prepare KDA payment (e.g., invalid wallet state).
ERR_KDA_FAILED: KDA transaction execution failed on-chain.
Important Notes
Security: For production apps, always verify the event.origin in your message listeners to ensure messages are coming from Metanet.page.
Tracking Requests: When sending a "pay" message that expects a response, always include a unique ref property. Use this ref to match incoming responses to your original requests.
App Assets: All apps are expected to have a logo.jpeg and a cover.jpeg image in their root directory. These are used for the app's icon and cover image in the platform UI.
Navigation Bar Layout: The platform's top navigation bar covers the top 70px of the screen. Your app design should account for this by providing an appropriate background behind the navigation area and ensuring all app content and interactive elements begin after the top 70px. Use padding-top: 70px on your main body content or margin-top: 70px for overlay elements. The background behind the navigation can be customized to match your app's design (white or any color of your choice).
Protocol is Everything: Remember that the JavaScript SDK is a convenience. The underlying postMessage protocol is the source of truth and can be implemented in any language or framework.
